# Exploration of Retrieval Augmented Generation
**Track Selected:** Track A, Text RAG (Short)

## Pipeline Summary
This project implements a fully local RAG pipeline that avoids API constraints. The vector database used is Chroma, saved to `./chroma_text`. A subset of the Wikipedia Simple English dataset (wikipedia/wikipedia (20220301.simple)) was embedded locally using the `all-MiniLM-L6-v2` sentence-transformer model. The generator uses the `llama-3.3-70b-instruct-awq` instruction-tuned model, accessed with a local/VPN endpoint, to create grounded answers from the top $k=4$ retrieved contexts.

## Reflection
### Part 1: Baseline Performance
All of the most simple and direct questions (Part 1: Q1, Q3, Q4, Q5, Q6, Q9) had exact answers. These questions, mostly "what is" questions, would almost certainly be answered in the first main paragraph of the article, which is what was extracted for this exercise. They were direct and to the point. For one question, the system was able to give a best answer despite not having the exact answer (Part 1: Q7). While it did not have the description for "Mario Brothers" specifically, it was still able to pull from related sources and give a good summary.  Notably, it did not note that Mario Bros is the same thing as Mario Brothers, but the article was still pulled up.

Due to the limited data extracted from each wiki article, the system was not able to answer a few questions about the specifics of a topic. This information was likely provided further down the articles (Part 1: Q2, Q8), which was not extracted for this exercise. Also, due to the limited number of sources overall (10,000), more obscure topics were not able to be answered (Part 1: Q10).

### Part 2: Custom Knowledge and Cross-Corpus Queries
For the added data, all the targeted questions were answered correctly (Part 2: Q1-5). In all cases, information that it did not specifically know was correctly answered with "I do not know", or with a best answer using the given data.

For example, in Part 2/Q6, it correctly stated that it did not have specific data for the question. It should be noted that my reference to the fictional city of Rivervale was not set apart from real locations of Rivervale, but it still considered both the real and fictional locations in its evaluation, all of which did not provide economic information. In Part 2/Q7, it was able to correctly relate the climate of the fictional land of Halas to the real-world equivalent of the Polar/Arctic climate. In Part 2/Q8, it correctly stated that it did not know the direct answer, but still made a good relation to the fictional food and who might have eaten it, giving a closest answer. Interestingly, it also noted another fictional food introduced to the data, finding the relation through the fictional world's data.

### Future Improvements
Given more time, I would experiment with extracting more data from the wiki dataset and use a chunking strategy with top_k to provide more data per article. I would also like to try observing the difference between chunking the data completely isolated versus having overlap between the chunks, just to see if ordering and context becomes easier or harder for the LLM to pick up. Additionally, increasing the top_k from 4 to 8 would provide a larger context window and may allow the model to draw up more complex conclusions.
